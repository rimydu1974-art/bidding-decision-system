## 二、架构原则

### 2.1 插拔式架构

**核心理念**：架构不变，升级只动一块，不影响其他功能

```
┌─────────────────────────────────────────┐
│              应用层 (App)                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ 前端页面 │ │ API路由  │ │ 管理后台 │   │
│  └─────────┘ └─────────┘ └─────────┘   │
├─────────────────────────────────────────┤
│              业务层 (Business)           │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ 决策引擎 │ │ 评分引擎 │ │ 风险引擎 │   │
│  └─────────┘ └─────────┘ └─────────┘   │
├─────────────────────────────────────────┤
│              核心层 (Core)               │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ 规则库   │ │ AI管道  │ │ 文档解析 │   │
│  └─────────┘ └─────────┘ └─────────┘   │
├─────────────────────────────────────────┤
│              基础层 (Infrastructure)     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ 数据库   │ │ 支付    │ │ 认证     │   │
│  └─────────┘ └─────────┘ └─────────┘   │
└─────────────────────────────────────────┘
```

### 2.2 稳定核心 + 外围升级

| 层级 | 策略 | 升级频率 |
|------|------|----------|
| **核心层** | 稳定，极少改动 | 季度/年度 |
| **业务层** | 模块化，独立升级 | 月度 |
| **应用层** | 快速迭代 | 周度 |

### 2.3 开源/闭源边界

| 类型 | 模块 | 策略 |
|------|------|------|
| **开源** | 项目框架、基础UI、认证 | 公开 |
| **闭源** | 规则库、AI提示词、支付、定价 | 严格保护 |
| **扩展** | 行业规则、知识库 | 可插拔 |

## 三、开发规范

### 3.1 代码规范

| 规范项 | 要求 |
|--------|------|
| **命名** | 驼峰命名（变量/函数）、PascalCase（组件/类型） |
| **注释** | 核心业务逻辑必须注释，复杂函数写JSDoc |
| **目录** | 按功能模块组织，避免跨模块引用 |
| **类型** | TypeScript严格模式，禁止any |

### 3.2 Git规范

| 类型 | 说明 |
|------|------|
| **分支** | main(生产) / develop(开发) / feature/*(功能) / hotfix/*(紧急) |
| **提交** | feat/fix/docs/refactor/test/chore |
| **版本** | 语义化版本：major.minor.patch |

### 3.3 安全规范

| 规范项 | 要求 |
|--------|------|
| **密钥** | 环境变量，禁止硬编码 |
| **权限** | 最小权限原则，RBAC |
| **数据** | 敏感数据加密，日志脱敏 |
| **API** | 限流、认证、输入校验 |

## 四、模块划分

### 4.1 开源模块（可公开）

```
├── 项目框架（Next.js路由结构）
├── 基础UI组件（components/ui/）
├── 认证系统（auth/）
├── 中间件（middleware.ts）
└── 工具函数（utils/）
```

### 4.2 闭源核心（严格保护）

```
├── 规则库（lib/rules/）          # 50条废标规则
├── AI提示词（lib/ai/prompts/）   # 结构化Prompt
├── 定价配置（lib/pricing.ts）    # 5层定价策略
├── 支付系统（lib/payment.ts）    # 微信/支付宝
├── 配额管理（lib/quota.ts）      # AI调用控制
├── 行为分析（lib/behavior.ts）   # 用户漏斗
└── 成本分析（admin/ai-cost/）   # API成本监控
```

### 4.3 扩展模块（可插拔）

```
├── 行业规则（prisma/IndustryRule）
├── 评分规则（prisma/ScoringRule）
├── 废标规则（prisma/WasteRule）
├── 案例中心（prisma/Case）
└── 知识库（prisma/KnowledgeItem）
```

## 六、升级规范

### 6.1 优先级定义

| 优先级 | 定义 | 响应时间 | 示例 |
|--------|------|----------|------|
| **P0** | 立即修复 | 24小时 | 系统崩溃、数据丢失 |
| **P1** | 紧急 | 3天 | 核心功能异常 |
| **P2** | 重要 | 2周 | 功能优化、小需求 |
| **P3** | 一般 | 1月 | 新功能、体验优化 |

### 6.2 升级流程

```
需求提出 → 评审 → 排期 → 开发 → 测试 → 上线
    │                    │
    └── 核心模块需审批   └── 外围模块快速迭代
```

### 6.3 核心模块升级审批

| 模块 | 审批人 | 要求 |
|------|--------|------|
| 规则库 | 产品负责人 | 影响分析报告 |
| AI提示词 | 技术负责人 | A/B测试验证 |
| 支付系统 | 财务+技术 | 安全审计 |
| 定价策略 | 产品+运营 | 商业影响评估 |
